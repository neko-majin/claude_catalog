はじめて（ゼロから設定して使う）

PCにくわしくない方でも、上から順にそのまま進めれば使えます。ターミナルという、文字で操作する画面を使います。下の図は実際の画面イメージです（◯印にマウスを乗せると用語の説明が出ます）。

ターミナル
文字を打ってPCに指示する画面。Windowsは「PowerShell」、Macは「ターミナル」。
コマンド
PCへの命令文。1行入力して Enter で実行します。
プロンプト
入力できる状態を示す記号（例: PS C:\> や % や $）。この右に打ちます。
パス
フォルダやファイルの「住所」。例: C:\Users\you\myapp
ディレクトリ
「フォルダ」と同じ意味。
リポジトリ
1つのプロジェクトのファイル一式（履歴付き）。略して「リポ」。
git
変更履歴を管理する仕組み。チーム開発の基盤。
コミット
変更内容を履歴として記録する操作（gitの基本）。
ブランチ
本流と別に作業を分ける「枝」。作業後に本流へ統合（マージ）する。
マージ
ブランチの変更を別のブランチに取り込むこと。
PR（プルリクエスト）
「この変更を取り込んで」という依頼。レビューしてからマージする。
CLI
コマンドで操作するツール（黒い画面で使うもの）。Claude Code のターミナル版もこれ。
IDE
コードを書く開発エディタ（VS Code など）。
CI / CD
テストやデプロイを自動で回す仕組み（例: GitHub Actions）。
API
プログラム同士が通信する窓口。APIキーで認証することが多い。
環境変数
PCやセッションに設定する値（APIキー等）。コードに直書きしない用途。
スキル
手順や知識をまとめた再利用部品（SKILL.md）。/名前 で呼べる。
プラグイン
スキルやコマンド等をまとめて配布する単位。/plugin で導入。
MCP
外部ツール・DB・APIに接続する仕組み（Model Context Protocol）。
フック(hook)
編集後に自動でlint等、決めた処理を強制実行する仕掛け。

STEP 1. ターミナルを開く

  • Windows: スタートメニューで「PowerShell」と入力して開く
  • Mac: 「ターミナル」と検索して開く（アプリ ＞ ユーティリティ にもあります）
Windows PowerShell
PS C:\Users\you> ← この行（プロンプト）が出ていれば準備OK
▲ 開いた直後の画面イメージ（Windows）。Macは you@mac ~ % のような表示です。

STEP 2. Claude Code をインストールする

下の1行をコピーして、ターミナルに貼り付けて Enter を押すだけです（貼り付け: Windowsは右クリック、Macは ⌘V）。

Windows PowerShell
PS C:\> irm https://claude.ai/install.ps1 | iex Downloading Claude Code... Installed successfully ✓
▲ 実行するとダウンロード→インストールが進み、最後に成功表示が出ます。

Mac / Linux はこちら:

curl -fsSL https://claude.ai/install.sh | bash

※「コマンドが見つからない」と出たら、ターミナルを一度閉じて開き直してください。

STEP 3. はじめて起動してログイン

claude と入力して Enter。初回はログインの案内が出るので、画面に従ってサインインします（Claudeの有料プラン等が必要）。

Claude Code
$ claude Welcome to Claude Code 👋 ? ログイン方法を選択 ▸ Claude account ブラウザで認証を完了してください…
▲ 起動直後。矢印キーで選び Enter → ブラウザで認証します。

STEP 4. 自分の作業フォルダで開く

cd は「フォルダを移動する」コマンドです。自分のプロジェクト（作業フォルダ）に移動してから claude を起動します。

Claude Code
$ cd C:\Users\you\myapp ← 作業フォルダのパス $ claude
▲ パスが分からないときは、フォルダをターミナルにドラッグ＆ドロップすると自動で入力されます。

STEP 5. お願いしてみる（普通の日本語でOK）

起動したら、やってほしいことを普通の文章で書くだけ。コマンドを覚える必要はありません。

Claude Code
> このプロジェクトに何が書いてあるか説明して このリポジトリは … を行うアプリです。 主な構成は src/（画面）, api/（通信）, tests/（テスト）です。 > ログイン画面のバグを直して 原因は … でした。修正して、テストも通しました ✓
▲ 会話するように頼むと、Claudeが調べて作業し、結果を返します。

STEP 6. 便利な基本操作

  • /help … 使えるコマンド一覧を表示
  • / を入力 … スキルやコマンドの候補が出る
  • 困ったら一度終了（Ctrl+C を2回）して claude で再起動

設定ファイルってどこ？（ざっくり）

  • ~/.claude/ … 自分専用の設定（どのプロジェクトでも効く）。~ は自分のホームフォルダの意味
  • プロジェクト内の .claude/ … そのプロジェクト用。gitにコミットすればチームで共有

細かい設定項目は「設定」タブ、各機能の詳しい使い方は上のカテゴリを見てください。

スキルとは？

SKILL.md に手順を書くと、Claude の“道具箱”に手順が追加されます。関連する場面で Claude が自動で使うほか、/スキル名 で直接呼び出せます。本体は使うときだけ読み込まれるので、普段は軽いままです。

使い方（呼び出し方）

  • 自動：説明(description)に合う依頼をすると Claude が自動で使う（例:「変更点を教えて」）
  • 手動：/スキル名 を入力して直接呼ぶ（例: /commit）

スキルの置き場所（＝誰が使えるか）

レベルパス使える範囲
個人 (Personal)~/.claude/skills/<name>/SKILL.md自分の全プロジェクト
プロジェクト (Project).claude/skills/<name>/SKILL.mdそのリポジトリのみ
プラグイン (Plugin)<plugin>/skills/<name>/SKILL.mdプラグインが有効な場所
管理 (Enterprise)管理設定組織内の全員

同名のときは 管理 > 個人 > プロジェクト の順で優先されます。

① ローカルで使う（自分だけ）

個人用フォルダにスキルを置くだけ。全プロジェクトで使えます。

mkdir -p ~/.claude/skills/my-skill # ~/.claude/skills/my-skill/SKILL.md を作成して手順を書く

② チームに共有する（リポジトリ単位）

プロジェクトの .claude/skills/ に置いて git にコミットします。リポジトリをクローンしたメンバー全員が同じスキルを使えます。

.claude/skills/<name>/SKILL.md ← コミットして共有

③ プラグインとして配布する（組織・複数リポジトリ）

skills/ を持つプラグインを作り、マーケットプレイス（git リポジトリ）で配布します。利用側は次の2コマンドで導入できます。

/plugin marketplace add <リポジトリ> /plugin install <プラグイン名>

組織全体へ一括配布したい場合は 管理設定 (Enterprise) でデプロイします。

SKILL.md の構成（frontmatter 早見）

スキルは SKILL.md 1ファイルが基本。先頭の --- で囲む YAML が設定、その下が指示本文です。

--- name: commit description: いつ使うか/何をするか（自動起動の判定に使われる・最重要） disable-model-invocation: true # 自動起動を止め /名前 でのみ実行 allowed-tools: Bash(git *) Read # 確認なしで使えるツール argument-hint: [issue-number] # 入力補完のヒント --- ここに手順や規約を書く（本文）
フィールド役割
description推奨。何を/いつ使うか。Claudeの自動起動判定に使用（最重要）
disable-model-invocationtrueで自動起動を禁止（手動 /名前 のみ）。副作用ある操作向け
user-invocablefalseで /メニューから非表示（背景知識用）
allowed-tools実行中に確認なしで使えるツール
argument-hint / arguments引数のヒント / 名前付き引数
context: fork / agentサブエージェントで分離実行する
model / effort / paths使用モデル / 努力レベル / 有効化するファイルパターン

引数を渡す（$ARGUMENTS）

本文中の $ARGUMENTS が呼び出し時の入力に置き換わります。位置指定は $0 $1（= $ARGUMENTS[0]…）。動的データは !`コマンド` で本文に埋め込めます（例: !`git diff HEAD`）。

--- name: fix-issue description: GitHubのissueを修正 --- GitHub issue $ARGUMENTS をコーディング規約に従って修正して

/fix-issue 123 → 「GitHub issue 123 を…」に展開されます。

サポートファイル

スキルはフォルダにでき、本文から参照する補助ファイル（テンプレート/例/スクリプト/詳細リファレンス）を同梱できます。SKILL.md は500行以内を目安に、詳細は別ファイルへ。

my-skill/ ├── SKILL.md # 本体（必須） ├── reference.md # 詳細（必要時に読み込み） └── scripts/ └── helper.py # 実行用（${CLAUDE_SKILL_DIR} で参照）

プラグインとは？

スキル・コマンド・エージェント・MCP などをまとめて配布する単位です。/plugin から導入・管理します。

プラグインに入れられるもの

1つのプラグインに複数のコンポーネントを同梱できます（plugin.json 以外はプラグイン直下に置く）。

場所内容
.claude-plugin/plugin.jsonマニフェスト（ここだけ .claude-plugin/ 内）
skills/<名前>/SKILL.mdスキル
commands/*.mdコマンド（旧形式・新規はskills推奨）
agents/カスタムサブエージェント
hooks/hooks.jsonフック
.mcp.json / .lsp.jsonMCP / LSP サーバー設定
monitors/monitors.jsonバックグラウンド監視
settings.json有効時のデフォルト設定

プラグインの作り方（最小3ステップ）

プラグインは「スキル等の中身 ＋ .claude-plugin/plugin.json」をまとめたフォルダです。1つのプラグインに skills / commands / agents / hooks / MCP(.mcp.json) / LSP(.lsp.json) をまとめて入れられます。

💬 かみくだき: マニフェスト(plugin.json)＝プラグインの名札（名前・説明・バージョンを書くファイル）。名前空間＝呼び出し名が /プラグイン名:スキル名 になる仕組みで、他のプラグインと名前がぶつからないための工夫です。

1. フォルダとマニフェストを作る

my-first-plugin/ ├── .claude-plugin/ │ └── plugin.json ← これだけ .claude-plugin/ の中 └── skills/ └── hello/ └── SKILL.md ← スキル本体（commands/ agents/ hooks/ も同じ階層）

.claude-plugin/plugin.json の最小例:

{ "name": "my-first-plugin", "description": "A greeting plugin to learn the basics", "version": "1.0.0", "author": { "name": "Your Name" } }

⚠ よくある間違い: skills/ commands/ agents/ hooks/ を .claude-plugin/ の中に入れないこと。中に入れるのは plugin.json だけで、他はプラグイン直下に置きます。

2. ローカルでテストする

claude --plugin-dir ./my-first-plugin # 起動後に /my-first-plugin:hello を実行して確認 # 中身を変えたら /reload-plugins で再読み込み（再起動不要）

プラグインのスキルは /プラグイン名:スキル名 のように名前空間付きで呼びます（他プラグインとの名前衝突を防ぐため）。
個人用にすぐ常設したいだけなら claude plugin init <名前> で ~/.claude/skills/ にひな形が作れます。

3. 配布する

マーケットプレイス（git リポジトリ）に載せると、利用側は次の2コマンドで導入できます。チーム限定にしたい場合はプライベートリポジトリでホストします。

/plugin marketplace add <リポジトリ> /plugin install <プラグイン名>

例: フック / エージェントを同梱する

hooks/hooks.json（編集後に lint を自動実行）:

{ "hooks": { "PostToolUse": [ { "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "jq -r '.tool_input.file_path' | xargs npm run lint:fix" }] } ] } }

agents/ にサブエージェント定義を置くと /agents に表示され、スキルの agent: やタスク委譲で使えます。MCPは .mcp.json を直下に置けば有効時に自動接続。

変更後は /reload-plugins で再読み込み。配布前に claude plugin validate で検証できます。

設定ファイルの使い分け（settings.json / settings.local.json）

ファイル用途git
.claude/settings.jsonチーム共有設定（全員に効く）コミットする
.claude/settings.local.json個人設定・実験用（自分だけ）コミットしない (gitignore)

同じ項目が両方にあるときは settings.local.json（個人）が優先されます。

settings.json の記述例

{ "permissions": { "allow": ["Bash(npm run test *)", "Read(~/.zshrc)"], "ask": ["Bash(git push *)"], "deny": ["Read(./.env)", "Read(./secrets/**)"], "defaultMode": "default" }, "env": { "FOO": "bar" } }
  • permissions.allow: 確認なしで許可するツール/コマンド
  • permissions.ask: 実行前に確認するもの（例: git push）
  • permissions.deny: 禁止（機密ファイルの読み取り防止など）
  • permissions.defaultMode: 既定の権限モード（default / acceptEdits など）
  • env: 全セッションに渡す環境変数

ほかに model（既定モデル）・hooks（イベント実行）・enabledPlugins（プラグインの有効化）なども指定できます。配列の設定（allow/deny など）はスコープをまたいでマージされます。

enabledPlugins の記述例

{ "enabledPlugins": { "formatter@acme-tools": true, "deployer@acme-tools": true, "analyzer@security-plugins": false } }

"プラグイン名@マーケットプレイス名": true/false の形式で、プラグインを個別に有効/無効化します。チーム共有の .claude/settings.json に書けば、リポジトリの全員で同じプラグイン構成を揃えられます。

MCP（外部ツール接続）の使い方

MCP（Model Context Protocol）は、Claude Code を外部のツール・DB・APIに接続する標準です。サーバーを追加すると、Claudeがそれらを直接読み書きできます（例:「JIRAの課題 ENG-4521 を実装してGitHubにPRを作って」）。

1. サーバーを追加する（claude mcp add）

接続方式は主に3種類。オプションはすべてサーバー名の前、-- 以降がローカル実行コマンドです。

# リモートHTTP（推奨） claude mcp add --transport http notion https://mcp.notion.com/mcp # 認証ヘッダー付き（例: GitHub） claude mcp add --transport http github https://api.githubcopilot.com/mcp/ \ --header "Authorization: Bearer YOUR_TOKEN" # ローカル stdio（手元でプロセス実行） claude mcp add --transport stdio --env AIRTABLE_API_KEY=KEY airtable \ -- npx -y airtable-mcp-server
💬 かみくだき: 接続方式(transport)は「つなぎ方」のこと。HTTP＝ネット上のサービスにつなぐ（多くはこれ）。stdio＝自分のPCで小さなプログラムを動かしてつなぐ。迷ったらサービス提供元のドキュメントのコマンドをそのままコピペでOK。

2. スコープ（保存先・共有範囲）

スコープ対象共有保存先
local（既定）そのプロジェクトのみ不可（個人）~/.claude.json
projectそのプロジェクト可（gitで共有）.mcp.json
user全プロジェクト不可（個人）~/.claude.json

優先順位は local > project > user > プラグイン > claude.ai。同名は最優先の定義をそのまま使用（フィールドはマージしない）。

3. チームで共有する（.mcp.json をコミット）

プロジェクトスコープで追加すると、リポジトリ直下に .mcp.json が作られます。これをコミットすれば全員が同じMCPを使えます（安全のため初回は承認を求められます）。

claude mcp add --transport http paypal --scope project https://mcp.paypal.com/mcp { "mcpServers": { "api-server": { "type": "http", "url": "${API_BASE_URL:-https://api.example.com}/mcp", "headers": { "Authorization": "Bearer ${API_KEY}" } } } }

※ ${VAR} / ${VAR:-既定値} で環境変数を展開でき、機密はコミットせず各自の環境変数で渡せます。

4. 使う・管理する

  • 追加後は普通に依頼するだけ（例:「過去24時間で最も多いエラーは？」）。
  • /mcp … 接続状況の確認・OAuth認証。
  • claude mcp list / claude mcp get <名前> / claude mcp remove <名前>
  • @サーバー:protocol://パス でリソース参照（例: @github:issue://123）。
  • /mcp__サーバー__プロンプト でサーバー提供のプロンプトをコマンド実行。

5. 公式リファレンスサーバー / 自作

filesystem / fetch / git / memory / sequential-thinking / time / everything はカタログの 「MCP」 タブを参照。サーバーの自作は公式 mcp-server-dev プラグインで雛形生成できます（/plugin install mcp-server-dev@claude-plugins-official）。

6. 組織で統制する

使えるMCPを組織で制限するには管理設定の allowedMcpServers / deniedMcpServers（managed-mcp）。プロジェクトの全MCPを一括承認するには設定 enableAllProjectMcpServers。

公式機能（チーム開発）の使い方

カテゴリ別に主要機能のセットアップ手順をまとめました。全機能の一覧と公式リンクはカタログの 「機能」 タブを参照してください。

CI・自動化

GitHub Actions — PR/Issueで @claude とメンションして実装・修正・PR作成

  1. ターミナルで claude を起動し /install-github-app を実行（リポジトリ管理者権限が必要。GitHubアプリ導入とSecrets設定をガイド）。
  2. API キーをリポジトリの Secrets に ANTHROPIC_API_KEY として追加。
  3. PR/Issue のコメントで @claude このバグを直して のようにメンション。

手動設定なら github.com/apps/claude を入れ、.github/workflows/ に下記を配置:

- uses: anthropics/claude-code-action@v1 with: anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }} # コメントの @claude に応答
💬 かみくだき: ワークフローファイル＝「PRが来たらこれを自動で実行」を書いた決まりファイル（.github/workflows/ に置く）。Secrets＝APIキーなどの秘密情報を安全に保管する場所（コードに直書きしない）。

Routines — 定期/イベントの無人実行（PCオフでも稼働）

  1. claude.ai/code/routines の New routine、または CLI で /schedule（例: /schedule daily PR review at 9am）。
  2. プロンプト・対象リポジトリ・環境・コネクタを設定。
  3. トリガーを選択: スケジュール / GitHubイベント(PR・Release) / API(HTTP POST)。複数併用可。

※ Web版Claude Code が有効で、Pro/Max/Team/Enterprise が必要。

💬 かみくだき: トリガー＝「何をきっかけに動かすか」。スケジュール＝決まった時刻に自動（例: 毎朝9時）。GitHubイベント＝PR作成などに反応。API＝他のツールから合図を送って起動。webhookはその合図を送る仕組みのこと。

Hooks — 編集後フォーマット等を強制実行

settings.json の hooks に「イベント・matcher・command」を定義。代表イベント: PreToolUse(ブロック可)/PostToolUse/SessionStart/Stop。

{ "hooks": { "PostToolUse": [ { "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/format.sh" }] } ] } }

/hooks で設定を確認できます。

その他: GitHub Code Review（全PRに自動レビュー）／scheduled-tasks（/loop でセッション内ポーリング）／GitLab CI/CD … 「機能」タブの公式リンク参照。

ガバナンス

  • permissions: settings.json の allow/ask/deny で実行/編集を制御。
